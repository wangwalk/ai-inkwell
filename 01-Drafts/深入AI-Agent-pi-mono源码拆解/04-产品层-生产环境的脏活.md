Agent 引擎只管循环，但对话断了要恢复、context 满了要压缩、API 限流要重试——谁来干？

上一篇我们看了 pi-ai 如何统一 20+ 家 LLM Provider。但你可能注意到一个问题：Agent Loop 是纯逻辑，它不知道磁盘、不知道 token 上限、不知道 429 错误。这些生产环境里逃不掉的脏活，全交给了 `AgentSession`。

## AgentSession：Agent 的生产环境外壳

先看它和 Agent 的分工：

| 职责 | Agent (pi-agent-core) | AgentSession (coding-agent) |
|------|----------------------|---------------------------|
| LLM 循环 | ✅ prompt → stream → toolUse → loop | ❌ |
| 事件系统 | ✅ subscribe/emit | ✅ 二次转发 + 扩展事件 |
| 消息持久化 | ❌ | ✅ 每条 message_end 写入 |
| Context 压缩 | ❌ | ✅ 自动触发 + 手动触发 |
| API 限流重试 | ❌ | ✅ 指数退避 |
| System Prompt | ❌ 只存字符串 | ✅ 动态拼装 |
| 会话恢复 | ❌ | ✅ JSONL + 树形分支 |

`AgentSession` 在构造时做一件关键的事——订阅 Agent 的事件流：

```typescript
// packages/coding-agent/src/core/agent-session.ts（简化）
constructor(config: AgentSessionConfig) {
  this.agent = config.agent;
  this.sessionManager = config.sessionManager;
  // 订阅 Agent 事件，驱动持久化、压缩、重试
  this._unsubscribeAgent = this.agent.subscribe(this._handleAgentEvent);
}
```

所有的"脏活"都挂在 `_handleAgentEvent` 这一个回调上。每次 `message_end` 事件触发，它同步写入 SessionManager；每次 `agent_end` 事件触发，它检查是否需要压缩或重试。

## 自动重试：指数退避 + 错误模式匹配

当 LLM 返回 error 时，AgentSession 先判断这个错误值不值得重试：

```typescript
// packages/coding-agent/src/core/agent-session.ts（简化）
private _isRetryableError(message: AssistantMessage): boolean {
  if (message.stopReason !== "error" || !message.errorMessage) return false;
  // context overflow 交给压缩处理，不走重试
  if (isContextOverflow(message, contextWindow)) return false;
  // 匹配十几种错误模式：overloaded、rate limit、429、500-504、connection error、fetch failed...
  return /overloaded|rate.?limit|too many requests|429|500|502|503|504|service.?unavailable|fetch failed|connection.?error/i
    .test(message.errorMessage);
}
```

两条设计决策值得注意：

1. **context overflow 和限流是两条独立的路径**。overflow 走压缩，限流走重试，互不干扰。
2. **错误判断用正则而不是错误码**。因为 20+ 家 Provider 的错误格式不统一，正则是最务实的选择。

重试本身是标准的指数退避：`baseDelayMs * 2^(attempt-1)`。错误消息从 Agent 内存中移除（避免污染下次 context），但保留在 session 文件里供调试。

## Context 压缩：找切割点 → LLM 摘要 → 替换历史

这是整个产品层最复杂的部分。当 context 接近 token 上限时，AgentSession 自动触发压缩。触发条件：

```typescript
// packages/coding-agent/src/core/compaction/compaction.ts（简化）
const DEFAULT_COMPACTION_SETTINGS = {
  enabled: true,
  reserveTokens: 16384,    // 为新对话预留的 token 数
  keepRecentTokens: 20000,  // 压缩后保留最近的 token 数
};

function shouldCompact(contextTokens, contextWindow, settings) {
  return contextTokens > contextWindow - settings.reserveTokens;
}
```

两种触发时机：**threshold**（用量接近上限）和 **overflow**（LLM 已经返回 context overflow 错误）。overflow 情况下会移除错误消息，压缩后自动重试。

### 找切割点

压缩不是把所有历史一刀切。算法从最新消息往回走，累积 token 数，直到达到 `keepRecentTokens`（默认 20000），然后在那个位置找一个合法切割点：

```typescript
// packages/coding-agent/src/core/compaction/compaction.ts（简化）
function findCutPoint(entries, startIndex, endIndex, keepRecentTokens) {
  // 从最新往回走，累积 token
  for (let i = endIndex - 1; i >= startIndex; i--) {
    if (entry.type !== "message") continue; // 跳过非消息 entry
    accumulatedTokens += estimateTokens(entry.message);
    if (accumulatedTokens >= keepRecentTokens) {
      // 找到最近的合法切割点
      cutIndex = closestValidCutPoint(i);
      break;
    }
  }
  return { firstKeptEntryIndex: cutIndex, isSplitTurn, turnStartIndex };
}
```

关键细节：**永远不会在 toolResult 处切割**。因为 toolResult 必须紧跟 toolCall，切断它们会让 LLM 困惑。合法的切割点包括 user、assistant、custom、bashExecution、branchSummary、compactionSummary 等类型。

如果切割点恰好落在一轮对话中间（比如 user 问了个问题，assistant 回复到一半就超限了），算法会识别出"split turn"，为被切断的前半段单独生成一份摘要。

### LLM 生成摘要

切割点找到后，被切掉的历史消息交给 LLM 生成结构化摘要：

```typescript
// packages/coding-agent/src/core/compaction/compaction.ts（简化）
async function compact(preparation, model, apiKey) {
  if (isSplitTurn) {
    // 并行生成：历史摘要 + turn 前缀摘要
    const [historySummary, turnPrefixSummary] = await Promise.all([
      generateSummary(messagesToSummarize, model, ...),
      generateTurnPrefixSummary(turnPrefixMessages, model, ...),
    ]);
    summary = `${historySummary}\n\n---\n\n**Turn Context (split turn):**\n\n${turnPrefixSummary}`;
  } else {
    summary = await generateSummary(messagesToSummarize, model, ...);
  }
  // 追加文件操作列表
  summary += formatFileOperations(readFiles, modifiedFiles);
  return { summary, firstKeptEntryId, tokensBefore };
}
```

摘要的 prompt 模板强制了一个固定格式：Goal → Constraints → Progress → Key Decisions → Next Steps → Critical Context。这不是随意的——结构化格式让后续的 LLM 调用能快速定位需要的上下文。

还有一个巧妙的设计：**迭代式摘要更新**。如果已经有过一次压缩，新的压缩不会从头总结全部历史，而是拿着上一次的摘要 + 新增消息做增量更新。这避免了"摘要的摘要的摘要"逐层失真。

Token 估算用的是 `chars / 4` 的粗略公式，故意偏保守（高估 token 数），宁可早压缩也不冒 overflow。

## 会话持久化：JSONL + 树形分支

SessionManager 用 JSONL 格式存储会话，每行一个 JSON 对象。第一行永远是 session header：

```jsonl
{"type":"session","version":3,"id":"abc-123","timestamp":"...","cwd":"/project"}
{"type":"message","id":"a1","parentId":null,"message":{"role":"user",...}}
{"type":"message","id":"b2","parentId":"a1","message":{"role":"assistant",...}}
{"type":"compaction","id":"c3","parentId":"b2","summary":"...","firstKeptEntryId":"a1"}
```

注意每个 entry 都有 `id` 和 `parentId`——这是一棵树，不是一个数组。

为什么要树？因为用户可以在任意消息处创建分支（比如"回到第 3 轮重新问"）。分支在同一个 JSONL 文件里追加，通过 `parentId` 指向分叉点。恢复会话时，从 leaf 沿着 `parentId` 往根走，收集路径上的所有 entry，就是当前分支的完整上下文。

```typescript
// packages/coding-agent/src/core/session-manager.ts（简化）
function buildSessionContext(entries, leafId) {
  // 从 leaf 往 root 走，收集路径
  const path = [];
  let current = byId.get(leafId);
  while (current) {
    path.unshift(current);
    current = current.parentId ? byId.get(current.parentId) : undefined;
  }
  // 路径上有 compaction？先输出摘要，再输出保留的消息
  // 路径上有 branch_summary？插入分支上下文
  return { messages, thinkingLevel, model };
}
```

还有一个实用的延迟写入优化：在第一条 assistant 消息到达之前，所有 entry 只在内存里攒着，不写磁盘。这样用户输入一条消息后直接退出，不会留下一堆只有用户消息的空 session 文件。

## System Prompt 的动态拼装

System Prompt 不是一个静态字符串，而是 AgentSession 每次都重新拼装的：

```typescript
// packages/coding-agent/src/core/system-prompt.ts（简化）
function buildSystemPrompt(options) {
  // 1. 基础人设 + 可用工具列表
  let prompt = `You are an expert coding assistant...
Available tools: ${toolsList}
Guidelines: ${guidelines}`;
  // 2. 追加自定义指令（appendSystemPrompt）
  prompt += appendSection;
  // 3. 追加项目级上下文文件（CLAUDE.md 等）
  for (const { path, content } of contextFiles) {
    prompt += `## ${path}\n\n${content}`;
  }
  // 4. 追加 Skills 描述
  prompt += formatSkillsForPrompt(skills);
  // 5. 追加时间和工作目录
  prompt += `\nCurrent date and time: ${dateTime}`;
  prompt += `\nCurrent working directory: ${cwd}`;
  return prompt;
}
```

拼装顺序有讲究：工具描述在最前面（LLM 需要最先知道自己能干什么），项目上下文在中间（可能很长），时间和目录在最后（每次都变）。

工具的 guidelines 也是动态生成的——如果你没启用 grep 工具，就不会出现"prefer grep over bash"的提示。这种条件生成避免了给 LLM 矛盾的指令。

## 设计点评

**做得好的地方：** 关注点分离非常清晰。Agent 是纯引擎，AgentSession 是生产适配层，SessionManager 是持久化层。三者职责不重叠，测试时可以独立 mock。压缩算法的"不切 toolResult"和"split turn 双摘要"体现了对 LLM 对话结构的深入理解。

**代价：** AgentSession 承担了太多职责——重试、压缩、持久化、事件转发、工具管理、扩展系统全在一个类里，文件超过 2000 行。`_handleAgentEvent` 一个回调驱动所有副作用，调试时需要在脑子里重建完整的事件流。这种"上帝回调"模式在功能不断迭代时会越来越难维护。

树形 JSONL 也有取舍。好处是追加写入极快、分支表达自然；代价是读取时需要全量解析再遍历，对超长会话有性能风险。
