Agent 在流式输出 Markdown，底部有个 spinner 在转，你拖了一下窗口——终端没闪，没乱，一切如常。

这不是理所当然的。终端没有 DOM，没有 Virtual DOM，没有局部重绘 API。你能做的只有：移动光标、清行、写字符。在这个约束下，pi-mono 自己搓了一套 TUI 框架，核心就一个目标——**只重绘变化的行**。

## 架构一览

整个 TUI 系统在 `packages/tui/src/` 下，三层结构：

```
Terminal (terminal.ts)      ← 终端抽象，管 stdin/stdout、光标、清屏
    ↑
TUI (tui.ts)                ← 差量渲染引擎，extends Container
    ↑
Components (components/)    ← Editor、Markdown、Loader、Overlay...
```

`TUI` 继承了 `Container`，而 `Container` 就是个简单的组件容器——遍历 children，收集每个组件的 `render(width)` 输出，拼成一个 `string[]`。每个字符串代表终端里的一行。

```typescript
// packages/tui/src/tui.ts（简化）
export interface Component {
  render(width: number): string[];
  handleInput?(data: string): void;
  invalidate(): void;
}
```

没有 props，没有 state management，没有 reconciliation。组件自己管自己的状态，render 时直接返回带 ANSI 转义码的字符串。简单粗暴，但够用。

## 差量渲染：三条路径

`doRender()` 是整个系统的核心，每次 `requestRender()` 触发一次。它拿到新的行数组后，和上一帧逐行比对，走三条路径之一：

**路径一：首次渲染**

```typescript
// packages/tui/src/tui.ts（简化）
if (this.previousLines.length === 0 && !widthChanged) {
  fullRender(false); // 直接输出，不清屏
  return;
}
```

没有上一帧，不需要比对，把所有行写出去就行。注意 `false` 参数——首次渲染不清屏，因为假设屏幕是干净的。

**路径二：宽度变化 / 内容收缩 → 全量重绘**

```typescript
if (widthChanged) {
  fullRender(true); // 清屏 + 全量输出
  return;
}
// 内容行数收缩 + 没有 overlay 时，全量重绘
if (this.clearOnShrink && newLines.length < this.maxLinesRendered && this.overlayStack.length === 0) {
  fullRender(true);
  return;
}
```

终端宽度变了意味着所有行的 word-wrap 都可能变，没法做差量，只能全量。`fullRender(true)` 会先清掉整个 scrollback 和屏幕（`\x1b[3J\x1b[2J\x1b[H`），再重新输出。

**路径三：正常更新 → 只刷变化的行**

这是最常见的路径，也是避免闪烁的关键：

```typescript
// packages/tui/src/tui.ts（简化）
let firstChanged = -1;
let lastChanged = -1;
for (let i = 0; i < maxLines; i++) {
  if (oldLine !== newLine) {
    if (firstChanged === -1) firstChanged = i;
    lastChanged = i;
  }
}

// 只重绘 firstChanged 到 lastChanged 之间的行
for (let i = firstChanged; i <= renderEnd; i++) {
  buffer += "\x1b[2K"; // 清当前行
  buffer += newLines[i];
}
```

找到第一行和最后一行变化的位置，光标移过去，只重写这个范围。Spinner 在转？只有 spinner 那一行变了，只刷一行。Markdown 在流式追加？只刷最后几行。上面已经渲染好的内容完全不动。

## CSI 2026：同步输出协议

光是差量还不够。如果你逐行写入终端，快的终端可能在你写到一半时就开始显示了，于是用户看到"半成品"——这就是闪烁的另一个来源。

pi-mono 用了一个不太知名的终端协议：**CSI 2026 Synchronized Output**。

```typescript
// packages/tui/src/tui.ts（简化）
let buffer = "\x1b[?2026h"; // 开始同步：终端暂停显示
// ... 把所有变化写入 buffer ...
buffer += "\x1b[?2026l"; // 结束同步：终端一次性刷新
this.terminal.write(buffer);
```

开始标记告诉终端"我要批量更新了，先别急着画"，结束标记告诉终端"好了，现在一次性显示"。这和浏览器的 `requestAnimationFrame` 思路一样——攒够一帧再画。

主流终端（iTerm2、Kitty、Windows Terminal、Ghostty 等）都支持这个协议。不支持的终端会忽略这两个转义码，行为退化到逐行刷新，不会出错。

## 组件体系

从 `packages/tui/src/index.ts` 的导出列表可以看出组件库的全貌：

| 组件 | 用途 |
|------|------|
| `Editor` | 多行文本编辑器，支持 word-wrap、undo/redo、kill-ring |
| `Markdown` | Markdown → ANSI 渲染，支持代码高亮、表格、引用 |
| `Loader` / `CancellableLoader` | Spinner 动画 |
| `SelectList` | 带模糊搜索的选择列表 |
| `Box` | 带边框的容器 |
| `Text` / `TruncatedText` | 文本显示 |
| `Image` | 终端图片（Kitty/iTerm2 协议） |
| `Spacer` | 占位 |

Overlay 系统值得单独说一下。`TUI.showOverlay()` 可以在基础内容之上叠加一个浮层（比如自动补全菜单、设置面板），支持 anchor 定位、百分比尺寸、margin 约束：

```typescript
// packages/tui/src/tui.ts（简化）
showOverlay(component: Component, options?: OverlayOptions): OverlayHandle {
  const entry = { component, options, preFocus: this.focusedComponent, hidden: false };
  this.overlayStack.push(entry);
  if (this.isOverlayVisible(entry)) this.setFocus(component); // 只有可见时才抢焦点
  // ...
}
```

Overlay 的合成发生在差量比对之前——`compositeOverlays()` 把浮层内容"画"到基础行上，然后整体参与差量比对。这意味着浮层的出现/消失也享受差量渲染，不需要全屏刷新。

## Markdown 组件：一个微型渲染器

`Markdown` 组件用 `marked` 库做 lexing，然后自己遍历 token 树生成带 ANSI 样式的字符串。

```typescript
// packages/tui/src/components/markdown.ts（简化）
render(width: number): string[] {
  if (this.cachedLines && this.cachedText === this.text && this.cachedWidth === width) {
    return this.cachedLines; // 文本和宽度都没变，直接返回缓存
  }
  const tokens = marked.lexer(normalizedText);
  // 遍历 tokens，生成带样式的行...
}
```

注意缓存策略：text 和 width 都没变才返回缓存。因为 width 变了意味着 word-wrap 结果不同，必须重新渲染。

它支持的 Markdown 元素很完整：heading、paragraph、code block（带语法高亮）、blockquote、list（支持嵌套）、table（自动列宽 + 自动换行）、link、inline code、bold/italic/strikethrough。表格渲染逻辑尤其复杂——要计算列宽、处理单元格内换行、画 Unicode 边框——占了 200 多行代码。

## CJK 与 IME：硬件光标的妥协

这是个很少被提及但很重要的细节。

在终端里输入中文或日文时，操作系统的 IME（输入法编辑器）需要知道当前光标的位置，才能在正确的地方弹出候选词窗口。但 pi-mono 的 TUI 在大多数时候是隐藏硬件光标的（`hideCursor()`），因为 cursor 闪烁会干扰 UI。

解决方案：`CURSOR_MARKER`。

```typescript
// packages/tui/src/tui.ts
export const CURSOR_MARKER = "\x1b_pi:c\x07"; // APC 序列，终端会忽略
```

当 Editor 组件获得焦点时，它在渲染输出的光标位置插入这个零宽度标记。TUI 在渲染时找到它、剥离它、然后用 CSI 序列把硬件光标移到那个位置：

```typescript
// packages/tui/src/tui.ts（简化）
private positionHardwareCursor(cursorPos, totalLines) {
  const rowDelta = targetRow - this.hardwareCursorRow;
  if (rowDelta > 0) buffer += `\x1b[${rowDelta}B`;
  else if (rowDelta < 0) buffer += `\x1b[${-rowDelta}A`;
  buffer += `\x1b[${targetCol + 1}G`; // 移到目标列
  this.terminal.write(buffer);
  if (this.showHardwareCursor) this.terminal.showCursor();
}
```

这样 IME 就能在正确的位置弹出候选词。默认情况下硬件光标不可见（`PI_HARDWARE_CURSOR=1` 可以打开），只为 IME 定位服务。

## 设计点评：为什么不用 Ink 或 Blessed？

Node.js 生态有现成的终端 UI 库——Ink（React for CLI）和 Blessed（curses 风格）。pi-mono 为什么要从头写？

我猜原因有几个：

**性能控制。** Ink 基于 React 的 reconciliation，每次渲染都要 diff Virtual DOM 然后映射到终端输出。pi-mono 的方案直接 diff 终端行字符串，没有中间层，路径更短。对于流式输出 Markdown 这种高频更新场景（每个 token 到达都触发渲染），这个差距是可感知的。

**Overlay 合成。** 在基础内容上叠加浮层（自动补全、设置面板）需要精确控制每一行的每一列。Ink 的布局模型基于 Yoga（Flexbox），不太适合这种"在第 3 行第 20 列插入一块内容"的需求。

**同步输出协议。** CSI 2026 需要在底层控制 write 时机，第三方框架很难插入这个逻辑。

**代价也很明显。** `tui.ts` 单文件超过 1100 行，光标位置计算、viewport 滚动、宽字符处理每一个都是细节地狱。`compositeLineAt()` 那个方法里甚至有一个 "belt-and-suspenders safeguard" 注释——作者自己都觉得宽度计算可能漂移。

这是典型的基础设施权衡：用更多的开发和维护成本，换来对渲染管线的完全掌控。对于一个每天被几十万开发者使用的 CLI 工具来说，这个选择是合理的。
