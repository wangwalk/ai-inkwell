把 OpenAI 换成 Claude，代码一行不改——这层抽象是怎么做到的？

上一篇我们拆了 Agent Loop，知道它最核心的动作就是调 `streamSimple`。这个函数来自最底层的 pi-ai 包，它只干一件事：给我一个 Model 和 Context，我帮你流式调 LLM。

但 LLM 不止一家。Anthropic 的 SSE 事件叫 `content_block_delta`，OpenAI 的叫 `response.output_text.delta`。Anthropic 的 stop reason 是 `end_turn`，OpenAI 的是 `completed`。Anthropic 的 thinking 是原生 block，OpenAI 要传 `reasoning_effort` 参数。

pi-ai 把这些差异全吃掉了。今天我们看看它怎么吃的，以及吃掉了什么。

## 插件式 Provider 注册

pi-ai 的 Provider 架构非常简洁。每个 API 协议注册两个函数就行：

```typescript
// packages/ai/src/api-registry.ts（简化）
interface ApiProvider<TApi extends Api> {
  api: TApi;
  stream: StreamFunction<TApi>;       // 原始调用，传 provider 专属选项
  streamSimple: StreamFunction<TApi>;  // 简化调用，传统一选项
}

const apiProviderRegistry = new Map<string, ApiProvider>();

function registerApiProvider(provider: ApiProvider): void {
  apiProviderRegistry.set(provider.api, provider);
}
```

注册时按 API 协议分类，不按厂商。OpenAI、xAI、Groq、OpenRouter 虽然是不同厂商，但都走 `openai-completions` 协议，共享同一个 stream 实现。内置 9 种协议覆盖 20+ 家 Provider——协议是 code，Provider 只是配置。

## 路由：一行代码的分发

上层调用 `streamSimple` 时，路由逻辑简单得离谱：

```typescript
// packages/ai/src/stream.ts（简化）
function streamSimple(model, context, options) {
  const provider = getApiProvider(model.api);  // 用 model.api 查注册表
  return provider.streamSimple(model, context, options);
}
```

`model.api` 就是路由键。一个 Model 对象自带 `api: "anthropic-messages"` 或 `api: "openai-responses"`，框架根据它找到对应的 stream 函数。上层代码完全不需要知道底下走的是什么协议。

## 统一事件流：Anthropic vs OpenAI 的差异抹平

这是整个抽象层最有价值的部分。不管底下是什么 Provider，上层收到的都是同一套事件：

```typescript
// packages/ai/src/types.ts（简化）
type AssistantMessageEvent =
  | { type: "start"; partial: AssistantMessage }
  | { type: "text_delta"; contentIndex: number; delta: string }
  | { type: "toolcall_start"; contentIndex: number }
  | { type: "toolcall_delta"; contentIndex: number; delta: string }
  | { type: "toolcall_end"; contentIndex: number; toolCall: ToolCall }
  | { type: "thinking_delta"; contentIndex: number; delta: string }
  | { type: "done"; reason: StopReason; message: AssistantMessage }
  | { type: "error"; reason: StopReason; error: AssistantMessage };
```

每个 Provider 的工作就是把自家的事件翻译成这套统一事件。看两个具体例子。

**Anthropic 的翻译逻辑**（核心片段）：

```typescript
// packages/ai/src/providers/anthropic.ts（简化）
for await (const event of anthropicStream) {
  if (event.type === "content_block_start") {
    if (event.content_block.type === "text") {
      stream.push({ type: "text_start", contentIndex: ... });
    } else if (event.content_block.type === "tool_use") {
      stream.push({ type: "toolcall_start", contentIndex: ... });
    }
  } else if (event.type === "content_block_delta") {
    if (event.delta.type === "text_delta") {
      stream.push({ type: "text_delta", delta: event.delta.text, ... });
    } else if (event.delta.type === "input_json_delta") {
      stream.push({ type: "toolcall_delta", delta: event.delta.partial_json, ... });
    }
  }
}
```

Anthropic 用 `content_block_start` / `content_block_delta` / `content_block_stop` 三段式。工具参数以 JSON 片段流式传入，Provider 内部用 `parseStreamingJson` 逐步解析。

**OpenAI Responses 的翻译逻辑** 走完全不同的事件模型——它用 `response.output_text.delta` 和 `response.function_call_arguments.delta` 这样的扁平事件。但翻译完之后，上层拿到的是一模一样的 `text_delta` 和 `toolcall_delta`。

StopReason 的映射也值得一看：

```typescript
// Anthropic: end_turn → stop, max_tokens → length, tool_use → toolUse
// OpenAI:    completed → stop (在 openai-responses-shared.ts 里处理)
```

五种统一 StopReason：`"stop" | "length" | "toolUse" | "error" | "aborted"`。上一篇说的 Agent Loop 就是靠 `toolUse` 判断要不要继续循环。

## Model 注册表：类型安全的模型描述

每个模型不只是一个 ID 字符串，而是一个完整的描述对象：

```typescript
// packages/ai/src/types.ts（简化）
interface Model<TApi extends Api> {
  id: string;               // "claude-opus-4-6-20250219"
  name: string;             // 展示名
  api: TApi;                // 路由键："anthropic-messages"
  provider: Provider;       // "anthropic"
  baseUrl: string;          // API 地址
  reasoning: boolean;       // 是否支持推理
  cost: { input, output, cacheRead, cacheWrite };  // $/百万 token
  contextWindow: number;
  maxTokens: number;
}
```

注意 `api` 字段的泛型约束——`Model<"anthropic-messages">` 和 `Model<"openai-responses">` 是不同类型。TypeScript 编译器会确保你不会把 Anthropic 模型传给 OpenAI 的 stream 函数。`api-registry.ts` 里的 `wrapStream` 还做了运行时校验：

```typescript
if (model.api !== api) {
  throw new Error(`Mismatched api: ${model.api} expected ${api}`);
}
```

编译期+运行时双重保险。

## Context 的可序列化设计

```typescript
interface Context {
  systemPrompt?: string;
  messages: Message[];
  tools?: Tool[];
}
```

Context 里全是 plain data，没有方法，没有类实例。这意味着它可以 `JSON.stringify` / `JSON.parse`，可以存数据库，可以跨进程传递。

更重要的是，它能跨 Provider 传递。你用 Claude 聊了 10 轮，想换成 GPT-5 继续——Context 不用改，只需要换一个 Model 对象。每个 Provider 内部的 `convertMessages` 函数负责把统一的 Message 格式翻译成自家 API 需要的格式。Anthropic 把 toolResult 包进 `role: "user"` 消息，OpenAI 用 `function_call_output` 类型——但这些都是 Provider 内部的事，Context 层不关心。

## 点评：最小公约数问题

这套抽象的收益很明显：上层代码（Agent Loop、UI）写一次，Provider 随便换。加一个新 Provider 只要实现两个函数。测试时可以 mock 整个 Provider 层。

但代价呢？

看 `OpenAICompletionsCompat` 这个类型，12 个兼容性开关：`supportsStore`、`supportsDeveloperRole`、`requiresThinkingAsText`、`requiresMistralToolIds`……这是"统一"抽象背后的真实复杂度。每个 Provider 都有自己的怪癖，抽象层要么抹平它，要么暴露一个开关让上层自己选。

还有一个更微妙的问题：Anthropic 的 `cache_control` 精确到单条消息级别，可以对 system prompt 和最后一条 user message 分别加缓存标记。OpenAI 的 prompt cache 只有全局开关。统一接口只能暴露 `cacheRetention: "none" | "short" | "long"` 三档，Anthropic 那边的细粒度控制就被抹平了。

这就是所有"统一抽象"的经典困境：你统一的是交集，不是并集。Provider 的独特能力要么被放弃，要么通过 `compat` 之类的逃生舱暴露出来，越积越多。

pi-ai 选择了务实路线——用 `StreamOptions` 传通用配置，用 `ProviderStreamOptions`（带 `Record<string, unknown>` 扩展）传特有配置，用两层函数（`stream` 接受 Provider 专属选项，`streamSimple` 只接受通用选项）让上层自己选抽象级别。

不完美，但能用。在 LLM 厂商 API 还在快速变化的今天，这可能就是最好的答案。
