# 深入 AI Agent — pi-mono 源码拆解：专栏设计

## 定位

跟着一个真实的开源 Agent 框架源码，搞懂 AI Agent 应用从 LLM 调用到产品落地的每一层设计。

## 目标读者

有 TypeScript 基础、用过 LLM API 但没深入过 Agent 框架内部的开发者。概念不跳过但不啰嗦，快速带过后重点放在源码分析和设计决策。

## 作者目标

- 通过源码拆解深入学习 Agent 应用架构
- 输出高质量技术内容，建立个人品牌
- 求职时展示独立学习能力和架构理解力（重点体现"读完有判断"而非"读过"）

## 写作角度

源码拆解。不是教程，不是从零构建，代码是用来说明设计的，不要求可运行。

## 每篇结构

1. **场景切入** — 从一个具体行为或问题引出这一层要解决什么
2. **架构拆解** — 核心抽象、关键源码片段、数据流
3. **设计点评** — trade-off 分析 + 作者自己的判断（好在哪、如果是自己会怎么做、潜在问题）
4. **思考题** — 留一个值得琢磨的问题

## 篇幅

每篇 3000-5000 字。

## 系列规划（6 篇，先打最强一拳）

### 第 1 篇：全景 — 一个 Agent 应用长什么样

**场景切入**：你在终端里输入一句话，Agent 读了文件、改了代码、跑了测试——背后经过了哪些模块？

**内容要点**：
- pi-mono 的 7 个包总览，哪些是核心、哪些是应用
- 三层依赖链：pi-ai ← pi-agent-core ← pi-coding-agent
- 一条用户输入从进入到响应的完整路径（走马观花，不深入）
- 点评：为什么要分三层而不是一个大包

### 第 2 篇：Agent 引擎 — 循环的心跳

**场景切入**：用户问一次，LLM API 可能被调了 5 次——Agent 和 Chatbot 的本质区别在这。

**内容要点**：
- agentLoop 的 while(true) 循环：调模型 → 收工具调用 → 执行工具 → 塞回结果 → 继续
- stopReason 驱动的状态机（end_turn / tool_use / error / aborted）
- 两层消息抽象：AgentMessage vs LLM Message，为什么需要两层
- Steering 和 FollowUp 队列：用户怎么在 Agent 运行中插话
- 点评：这个循环设计的简洁之处和潜在瓶颈

### 第 3 篇：LLM 抽象层 — 统一 20+ 提供商的代价

**场景切入**：把 OpenAI 换成 Claude，代码一行不改——这层抽象是怎么做到的？

**内容要点**：
- Provider 插件架构：每个提供商一个 stream 函数 + 消息转换器
- 统一事件流（text_delta / toolcall_start / done）
- Model 注册表：类型安全的模型描述
- Context 的可序列化设计：为什么跨 provider 传递很重要
- 点评：统一抽象的收益与"最小公约数"问题

### 第 4 篇：产品层 — 生产环境的脏活

**场景切入**：Agent 引擎只管循环，但对话断了要恢复、context 满了要压缩、API 限流要重试——谁来干？

**内容要点**：
- AgentSession 包装层：重试、压缩、持久化、事件转发
- Context 压缩策略：找切割点 → LLM 生成摘要 → 替换历史
- 会话持久化：JSONL + 树形分支
- System Prompt 的动态拼装
- 点评：关注点分离做得好的地方和复杂度的代价

### 第 5 篇：终端 UI — 无闪烁的差量渲染

**场景切入**：Agent 在流式输出 Markdown、同时底部有个 spinner 在转——终端怎么做到不闪不乱？

**内容要点**：
- 差量渲染策略：只重绘变化的行
- CSI 2026 同步输出协议
- 组件体系：Editor、Markdown、Overlay 等
- CJK 输入的 IME 处理
- 点评：用终端做复杂 UI 的设计取舍

### 第 6 篇：插件系统 — 不改源码扩展一切

**场景切入**：想给 Agent 加一个"搜索网页"的能力，不 fork 源码能做到吗？

**内容要点**：
- ExtensionAPI：注册工具、命令、快捷键、事件监听
- 生命周期钩子：before_agent_start / tool_call / agent_end
- 权限门控和路径保护
- Pi Packages 分发机制
- 点评：插件系统的边界在哪里，什么该内置什么该插件化
