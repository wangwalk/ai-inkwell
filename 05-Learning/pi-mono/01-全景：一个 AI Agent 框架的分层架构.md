# 全景：一个 AI Agent 框架的分层架构

> 《深入 AI Agent 引擎 — Pi Mono》第 01 篇

---

你让 AI 帮你改一段代码，它读了文件、跑了测试、修了 bug、再跑一遍测试确认通过——整个过程你只说了一句话。

这背后不是一个 API 调用，而是一个**循环**：调模型 → 执行工具 → 把结果喂回模型 → 再调模型……直到任务完成。这个循环就是 Agent 的核心，也是 Agent 和普通 Chatbot 的分界线。

Pi Mono 是一个完整的 AI Agent 框架。它不是某个模型的 SDK 封装，而是从 LLM 调用到终端渲染、从会话管理到插件系统，全部自己实现的垂直技术栈。本篇带你建立对它的全景认知。

---

## 一个 Monorepo，七个包

Pi Mono 采用 npm workspaces 管理，所有包共享同一个版本号（lockstep versioning），当前 v0.63.0。七个包各司其职：

| 包 | 一句话定位 |
|---|-----------|
| **pi-ai** | LLM 统一接口：用一套 API 调 20+ 模型提供商 |
| **pi-tui** | 终端 UI 引擎：差分渲染、组件系统、图片显示 |
| **pi-agent-core** | Agent 运行时：循环控制、工具执行、事件驱动 |
| **pi-coding-agent** | 交互式编程助手：完整的 CLI 产品 |
| **pi-web-ui** | Web 组件库：浏览器端的聊天界面 |
| **pi-mom** | Slack Bot：能自我管理的 AI 助手 |
| **pi-pods** | GPU 部署工具：在远程机器上跑开源模型 |

七个包不是平级的。它们有清晰的层次关系。

## 三层架构

如果只记住一张图，记这张：

```
┌─────────────────────────────────────────────┐
│              产品层 (Product)                │
│                                             │
│   pi-coding-agent   pi-mom   pi-web-ui      │
│        完整的用户交互产品                      │
└──────────────────────┬──────────────────────┘
                       │ 依赖
┌──────────────────────▼──────────────────────┐
│            引擎层 (Engine)                   │
│                                             │
│             pi-agent-core                   │
│      Agent 循环、工具执行、事件系统             │
└──────────────────────┬──────────────────────┘
                       │ 依赖
┌──────────────────────▼──────────────────────┐
│            基础层 (Foundation)               │
│                                             │
│          pi-ai            pi-tui            │
│     LLM 统一接口       终端渲染引擎            │
└─────────────────────────────────────────────┘
```

**基础层**解决两个独立问题：怎么调模型（pi-ai），怎么画界面（pi-tui）。两者互不依赖。

**引擎层**只依赖 pi-ai，不关心 UI。它定义了 Agent 的核心行为：调模型 → 模型说要用工具 → 执行工具 → 把结果送回模型 → 循环。这个循环是所有上层产品共享的。

**产品层**组合下面两层，做成面向用户的东西。pi-coding-agent 同时依赖 pi-ai、pi-agent-core、pi-tui 三个包；pi-web-ui 用 pi-ai 调模型，但渲染走浏览器而非终端；pi-mom 甚至直接依赖 pi-coding-agent，把整个编程助手作为 Slack 消息的处理后端。

层次的价值在于：**每一层都可以独立使用**。你可以只用 pi-ai 做模型调用，不需要 Agent；可以只用 pi-agent-core 做后端 Agent，不需要任何 UI；也可以在 pi-agent-core 之上搭一个完全不同的产品。

## 为什么不用 LangChain？

这是读源码时绕不开的问题。市面上已经有 LangChain、Vercel AI SDK 等成熟框架，为什么要自己写？

从代码中能看到的设计取舍：

**控制力换来的代价是工作量，但换来的收益是精确。**

Pi Mono 的 LLM 抽象层（pi-ai）直接对接每个提供商的原始 SDK——Anthropic 的、OpenAI 的、Google 的、AWS Bedrock 的。它不依赖任何中间抽象层。这意味着：

- **流式行为完全可控**：partial JSON 解析、token 计数、事件格式，都是自己定义的，不受第三方 SDK 版本更新影响
- **Provider 特性不被抹平**：Anthropic 的 Prompt Cache、OpenAI 的 Response API、Google 的 Thinking 模式，都能精确暴露，而不是降级到最大公约数
- **工具调用的执行权在自己手里**：Agent 引擎决定何时执行、是否执行、并行还是串行，没有框架替你做这个决策

代价是什么？每接入一个新的提供商，都得写一个完整的 Provider 适配。但 Pi Mono 选择了这条路，因为 Agent 的核心体验——响应速度、工具调用的可靠性、上下文管理的精细度——都取决于这些底层细节。

## 依赖链如何反映设计意图

看构建顺序就能理解架构思路：

```
tui → ai → agent-core → coding-agent → mom → web-ui → pods
```

先构建两个互不依赖的基础（tui 和 ai），然后构建引擎（agent-core），再构建各个产品。这不是随意安排的——它精确反映了依赖方向。

几个值得注意的依赖关系：

**pi-agent-core 不依赖 pi-tui。** Agent 引擎是纯逻辑的，不关心怎么显示。这让同一个 Agent 可以在终端（pi-coding-agent）、浏览器（pi-web-ui）、Slack（pi-mom）中运行，只换 UI 层。

**pi-mom 依赖 pi-coding-agent，而非只依赖 pi-agent-core。** 这说明 Mom 不是从头造一个 Agent，而是把已有的编程助手整个拿来，包上 Slack 的壳。这是一个务实的选择——复用已有的工具集（bash、read、write、edit），不重复造轮子。

**pi-pods 只依赖 pi-agent-core。** 它是一个轻量级的基础设施管理工具，只需要 Agent 循环来交互式管理 GPU 机器，不需要编程助手的全套工具。

## 关注点分离的三个层面

Pi Mono 的分层不只是代码组织，它在三个层面上实践了关注点分离：

### 模型调用 vs Agent 行为

pi-ai 只管"怎么调模型"：发请求、解析流、统计 token。它不知道什么是工具执行，不知道什么是对话循环。这些都是 pi-agent-core 的事。

这意味着 pi-ai 可以用在任何需要调 LLM 的场景——不一定是 Agent。简单的一次性问答、批量翻译、文本分类，都可以直接用 pi-ai，不需要引入 Agent 的复杂性。

### 引擎逻辑 vs 产品逻辑

pi-agent-core 的 Agent 是一个纯循环引擎。而实际产品需要的重试、压缩、持久化、权限控制，都在 pi-coding-agent 的 AgentSession 层处理。

引擎保持纯粹，产品层叠加复杂性。这让引擎层可以被不同产品以不同方式使用——pi-coding-agent 需要会话持久化，pi-pods 可能不需要。

### 渲染 vs 逻辑

pi-tui 和 pi-agent-core 完全平行。终端怎么画和 Agent 怎么想是两件独立的事。pi-web-ui 证明了这一点——它用完全不同的渲染方式（Web Components），但复用了相同的模型调用和 Agent 逻辑。

## 七个包，四种角色

换个角度看这七个包，它们其实扮演四种角色：

```
基础设施    pi-ai (模型调用)、pi-tui (终端渲染)
运行时      pi-agent-core (Agent 引擎)
产品        pi-coding-agent (CLI)、pi-web-ui (浏览器)、pi-mom (Slack)
工具        pi-pods (GPU 部署)
```

**基础设施**提供原子能力，不做业务假设。

**运行时**定义 Agent 的行为模型，但不绑定具体产品形态。

**产品**面向具体场景和用户，组合下层能力。

**工具**是独立的辅助，不被其他包依赖。

## 小结

Pi Mono 的架构选择可以归结为一句话：**垂直整合，水平解耦**。

垂直整合——从 LLM API 调用到终端像素渲染，不依赖外部框架，每一层自己掌控。这给了它在流式响应、工具调用、上下文管理上的精细控制力。

水平解耦——同一层的包互不依赖（pi-ai 和 pi-tui），不同产品可以自由组合底层能力（CLI、Web、Slack 共享同一个引擎）。
