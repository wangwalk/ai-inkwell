想给 Agent 加一个"搜索网页"的能力，不 fork 源码能做到吗？

能。pi-mono 的插件系统允许你在不碰一行源码的情况下，注册工具、拦截事件、注入 UI 组件，甚至替换模型提供商。这是整个系列的最后一篇，我们来看看这套扩展机制是怎么设计的。

## 一个插件长什么样

一个 pi 插件就是一个 TypeScript 文件，默认导出一个工厂函数：

```typescript
// .pi/extensions/web-search.ts（简化）
import { Type } from "@sinclair/typebox";
import type { ExtensionAPI } from "@mariozechner/pi-coding-agent";

export default function (pi: ExtensionAPI) {
  // 注册一个 LLM 可调用的工具
  pi.registerTool({
    name: "web_search",
    label: "Web Search",
    description: "Search the web for information",
    parameters: Type.Object({
      query: Type.String({ description: "Search query" }),
    }),
    async execute(toolCallId, params, signal, onUpdate, ctx) {
      const result = await fetch(`https://api.example.com/search?q=${params.query}`);
      return { content: [{ type: "text", text: await result.text() }] };
    },
  });

  // 监听事件
  pi.on("agent_end", async (event, ctx) => {
    ctx.ui.notify("Agent 完成了一轮对话");
  });
}
```

工厂函数接收一个 `ExtensionAPI` 对象，所有能力都通过它注册。没有类继承，没有装饰器，就是函数调用。

## ExtensionAPI：五类能力

`ExtensionAPI` 提供的方法可以分成五类：

| 类别 | 方法 | 作用 |
|------|------|------|
| 事件监听 | `on(event, handler)` | 监听十余种生命周期事件 |
| 工具注册 | `registerTool(definition)` | 注册 LLM 可调用的工具 |
| 命令/快捷键 | `registerCommand()` / `registerShortcut()` | 注册斜杠命令和键盘快捷键 |
| 运行时操作 | `sendMessage()` / `setModel()` / `exec()` 等 | 主动控制 Agent 行为 |
| 提供商注册 | `registerProvider(name, config)` | 注册或覆盖模型提供商 |

注意一个细节：注册类方法（registerXxx）在加载时就执行，写入 Extension 对象；而操作类方法（sendMessage 等）委托给共享的 `ExtensionRuntime`，在 Agent 会话真正初始化后才可用。加载阶段调用操作方法会直接抛错：

```typescript
// packages/coding-agent/src/core/extensions/loader.ts（简化）
export function createExtensionRuntime(): ExtensionRuntime {
  const notInitialized = () => {
    throw new Error("Extension runtime not initialized.");
  };
  return {
    sendMessage: notInitialized,
    sendUserMessage: notInitialized,
    // ... 所有操作方法都指向 throw
    flagValues: new Map(),
    pendingProviderRegistrations: [],
  };
}
```

这个设计把"声明"和"执行"分开了——加载时只做声明，运行时才注入真正的实现。还记得[第一篇](./01-全景-一个Agent应用长什么样.md)提到 `createAgentSession` 里加载 extensions 吗？就是在那里通过 `bindCore()` 把真实方法绑上去的。

## 生命周期钩子：20+ 个切入点

事件系统是插件的核心能力。看看有哪些钩子：

| 事件 | 触发时机 | 能做什么 |
|------|----------|---------|
| `session_start` | 会话初始化 | 初始化插件状态 |
| `before_agent_start` | 用户发消息后、Agent 循环前 | 注入额外消息、修改 system prompt |
| `agent_start` / `agent_end` | Agent 循环开始/结束 | 统计、日志 |
| `turn_start` / `turn_end` | 每轮对话开始/结束 | 追踪轮次 |
| `context` | 每次 LLM 调用前 | 修改发给 LLM 的消息列表 |
| `tool_call` | 工具执行前 | **拦截并阻止**工具执行 |
| `tool_result` | 工具执行后 | **修改**工具返回结果 |
| `input` | 用户输入到达后 | 变换输入文本、完全接管处理 |
| `user_bash` | 用户执行 bash 命令 | 替换执行逻辑 |
| `resources_discover` | 启动/重载时 | 提供额外的 skill/prompt/theme 路径 |
| `session_before_*` | 切换/fork/compact/tree 前 | **取消**操作 |
| `model_select` | 模型切换时 | 响应模型变化 |
| `session_shutdown` | 进程退出 | 清理资源 |

关键设计：不同事件的处理逻辑不同。`ExtensionRunner` 里有多个专用的 emit 方法：

```typescript
// packages/coding-agent/src/core/extensions/runner.ts（简化）
// 通用 emit —— 顺序执行，session_before_* 支持取消
async emit(event) {
  for (const ext of this.extensions) {
    for (const handler of ext.handlers.get(event.type) ?? []) {
      const result = await handler(event, ctx);
      if (result?.cancel) return result; // 短路
    }
  }
}

// tool_call 专用 —— 支持 block
async emitToolCall(event) {
  const result = await handler(event, ctx);
  if (result?.block) return result; // 阻止工具执行
}

// tool_result 专用 —— 链式修改（currentEvent 是累积器）
async emitToolResult(event) {
  const currentEvent = { ...event }; // 可变副本
  // 每个 handler 可以修改 content/details/isError
  if (result.content !== undefined) currentEvent.content = result.content;
  // 最终返回累积后的 currentEvent
  return { content: currentEvent.content, details: currentEvent.details, isError: currentEvent.isError };
}

// input 专用 —— 支持 transform 和 handled（文本和图片都可变换）
async emitInput(text, images, source) {
  const result = await handler(event, ctx);
  if (result?.action === "handled") return result; // 完全接管
  if (result?.action === "transform") {
    currentText = result.text;
    currentImages = result.images ?? currentImages; // 图片也可变换
  }
}
```

不是所有事件都走同一条路径。`tool_result` 是链式的（每个 handler 可以修改结果，修改后传给下一个），`tool_call` 是短路的（一个 handler 说 block 就停），`input` 支持三种模式（continue / transform / handled）。

这比"所有事件都 emit 一下"要复杂，但换来的是精确的语义——你能明确知道自己的 handler 返回值意味着什么。

## 工具包装：拦截内置工具

[第四篇](./04-产品层-生产环境的脏活.md)提到 `_handleAgentEvent` 会把事件分发给插件。具体到工具层面，靠的是 `wrapper.ts` 里的包装：

```typescript
// packages/coding-agent/src/core/extensions/wrapper.ts（简化）
export function wrapToolWithExtensions(tool, runner) {
  return {
    ...tool,
    execute: async (toolCallId, params, signal, onUpdate) => {
      // 1. 执行前：触发 tool_call，可以 block
      if (runner.hasHandlers("tool_call")) {
        const callResult = await runner.emitToolCall({
          type: "tool_call", toolName: tool.name, toolCallId, input: params,
        });
        if (callResult?.block) {
          throw new Error(callResult.reason || "Blocked by extension");
        }
      }

      // 2. 执行工具
      const result = await tool.execute(toolCallId, params, signal, onUpdate);

      // 3. 执行后：触发 tool_result，可以修改返回值
      if (runner.hasHandlers("tool_result")) {
        const modified = await runner.emitToolResult({
          type: "tool_result", toolName: tool.name, content: result.content, ...
        });
        if (modified) return { content: modified.content ?? result.content, ... };
      }

      return result;
    },
  };
}
```

每个内置工具（bash、read、edit、grep...）都会被这层包装。所以你不需要替换工具实现，只需要监听 `tool_call` 就能做权限门控：

```typescript
pi.on("tool_call", async (event) => {
  if (event.toolName === "bash" && event.input.command.includes("rm -rf")) {
    return { block: true, reason: "危险命令被拦截" };
  }
});
```

## 快捷键保护：有些键不让碰

插件可以注册键盘快捷键，但不是所有键都能覆盖。`runner.ts` 维护了一个保留列表：

```typescript
const RESERVED_ACTIONS_FOR_EXTENSION_CONFLICTS: ReadonlyArray<KeyAction> = [
  "interrupt", "clear", "exit", "suspend",
  "cycleThinkingLevel", "cycleModelForward", "cycleModelBackward",
  "selectModel", "expandTools", "toggleThinking",
  "externalEditor", "followUp", "submit",
  "selectConfirm", "selectCancel", "copy", "deleteToLineEnd",
];
```

`interrupt`（Ctrl+C）、`exit`（Ctrl+D）这些关键操作不允许被插件覆盖。非保留快捷键可以被插件接管，但如果多个插件注册了同一个键，会产生冲突警告，后加载的赢。

命令也是同理——如果插件注册的命令名和内置命令冲突，直接跳过并警告。

## 加载机制：从哪里找到插件

`loader.ts` 里的 `discoverAndLoadExtensions` 定义了三级发现顺序：

```
1. 全局插件：~/.pi/extensions/
2. 项目插件：{cwd}/.pi/extensions/
3. 配置文件中显式指定的路径
```

每个目录下支持三种形式：
- **直接文件**：`extensions/my-tool.ts` — 直接加载
- **目录 + index**：`extensions/my-tool/index.ts` — 加载入口
- **目录 + package.json**：`extensions/my-tool/package.json` 里声明 `pi.extensions` 字段 — 按声明加载

```json
{
  "name": "my-pi-extension",
  "pi": {
    "extensions": ["src/main.ts"],
    "themes": ["themes/dark.ts"],
    "skills": ["skills/review.md"]
  }
}
```

`pi` 字段不止能声明 extensions，还能声明 themes、skills、prompts。这就是 Pi Packages 的分发机制——一个 npm 包可以同时提供插件代码、主题和 skill 文件。

加载用的是 `jiti`（一个 TypeScript 即时编译器），所以插件不需要预编译，直接写 `.ts` 就能用。在 Bun 编译后的二进制里，通过 `virtualModules` 把依赖注入给 jiti，保证插件能 import `@mariozechner/pi-coding-agent` 等包。

## 设计点评

这套插件系统的边界在哪？

**做得好的**：事件粒度足够细，20+ 个钩子覆盖了 Agent 生命周期的关键节点。`tool_call` 拦截 + `tool_result` 修改的组合非常灵活，基本上能实现任何工具层面的定制。快捷键和命令的冲突保护也很务实。

**值得注意的**：所有 handler 是顺序执行的，没有优先级机制。如果两个插件都监听 `tool_call` 并且都想 block，谁先加载谁赢。全局插件先于项目插件加载，所以全局插件的 handler 优先级更高——这是隐式的，没有文档说明。

**什么该插件化，什么该内置？** pi-mono 的选择很明确：核心工具（bash、read、edit、grep、write、find、ls）内置，其他一切都可以通过插件扩展。模型提供商可以插件化（`registerProvider`），UI 组件可以插件化（`setWidget`、`setFooter`、`setEditorComponent`），甚至输入处理都可以插件化（`input` 事件）。但 Agent 循环本身——[第二篇](./02-Agent引擎-循环的心跳.md)讲的 while 循环——不能被替换，只能被 hook。

这是一条清晰的线：**流程可以被观察和干预，但不能被替换**。
